# 智慧学生管理系统 — Agent 开发指导手册

> 本文档指导 AI Agent 逐步构建本系统。**核心原则：每一步都要小步推进、确认后再继续，绝不一次性写完所有代码然后反复修 bug。**

---

## 铁律（每次对话开始时必读）

1. **一次只做一个阶段**，当前阶段未完成并得到用户确认前，禁止进入下一阶段
2. **视觉决策必须先出原型**：配色、布局、组件样式等，先用一个独立 HTML 文件做出可预览的效果，让用户看到实际样子后再写入项目
3. **主动提问，不要猜**：遇到设计选择（配色、图标风格、交互方式、图片/插画风格）必须停下来问用户，列出 2-3 个选项并说明各自优劣
4. **有好方案要主动说**：如果你有更优的实现思路，直接提出来让用户决定，不要闷头按默认方案干
5. **每个阶段结束时汇报**：说明做了什么、有什么问题、下一步计划，等用户确认再继续

---

## 项目概览

- **项目类型**：前后端分离的全栈 Web 应用
- **技术栈**：Vue 3 + TypeScript + Element Plus / Python FastAPI / SQLite（开发）/ SQLAlchemy / JWT / GSAP
- **AI 模块**：后期建设，核心功能稳定后再接入 DeepSeek API
- **三种角色**：学生端、教师端、管理端（共用一套导航栏组件）
- **详细功能清单**：参见 `智慧学生管理系统-PRD-V1.3.docx`

---

## 阶段规划

### 阶段 0：环境搭建与项目初始化

**目标**：跑通前后端空项目，确认开发环境正常。

**步骤**：
1. 初始化前端项目：`npm create vite@latest frontend -- --template vue-ts`
2. 安装基础依赖：`element-plus`、`vue-router`、`pinia`、`axios`、`gsap`
3. 初始化后端项目：创建 `backend/` 目录，建立 FastAPI 项目结构
4. 安装后端依赖：`fastapi`、`uvicorn`、`sqlalchemy`、`pydantic`、`python-jose`、`bcrypt`、`alembic`
5. 创建 SQLite 数据库，配置连接
6. 前后端分别跑起来，确认能访问

**检查点**：
- [ ] 前端 `http://localhost:5173` 能看到空白页
- [ ] 后端 `http://localhost:8000/docs` 能看到 Swagger 页面
- [ ] 用户确认后再进入阶段 1

---

### 阶段 1：设计系统（视觉基础）

**目标**：确定全局配色、字体、组件风格。**这步必须先出 HTML 原型让用户看效果。**

#### 1.1 配色方案

**操作**：生成一个独立的 `color-preview.html` 文件，包含 2-3 套配色方案的实际效果展示：

- 方案 A：冷色系（深蓝 #2B579A + 浅灰 #F5F7FA）— 专业沉稳，教育类常见
- 方案 B：暖色系（青绿 #10B981 + 米白 #FFFBEB）— 清新友好，适合中小学
- 方案 C：中性色（靛紫 #6366F1 + 浅灰 #F9FAFB）— 现代科技感

每套方案展示：导航栏、按钮、卡片、表格、表单的实际渲染效果。

**交互要求**：把 HTML 文件给用户看，让用户选一个方案（或提出修改意见），确认后记录到项目中。

#### 1.2 字体与排版

**操作**：确认中文字体栈（推荐 `system-ui, "Microsoft YaHei", "PingFang SC", sans-serif`），确定标题/正文/辅助文字的字号层级。

#### 1.3 图标方案

**提问**：图标用 Element Plus 内置图标，还是引入 Lucide / Remix Icon？是否有图标风格偏好（线性 / 填充）？

#### 1.4 图片与插画

**提问**：空状态页面（如无成绩、无作业）用什么风格的插画？
- 选项 A：简约线条插画（类似 unDraw 风格）
- 选项 B：纯色块扁平插画
- 选项 C：不用插画，用图标+文字提示即可

**检查点**：
- [ ] 配色方案已确认
- [ ] 字体层级已确认
- [ ] 图标方案已确认
- [ ] 插画风格已确认
- [ ] 将确定的设计 token 写入前端项目的 `src/styles/variables.scss`
- [ ] 用户确认后再进入阶段 2

---

### 阶段 2：导航栏与布局框架

**目标**：实现共用的 GSAP 滑动导航栏 + 页面基础布局。

#### 2.1 导航栏组件

**先出原型**：生成一个独立的 `nav-prototype.html`，实现：
- 顶部导航栏：Logo（左）+ nav-items（居中）+ 用户信息（右）
- GSAP Flip 滑动指示条效果
- 用假数据展示学生端（学习/生活/我的）和教师端（教学/管理/我的）的切换效果

**让用户在浏览器中实际操作确认**，满意后再写入项目。

#### 2.2 页面布局

实现 `AppLayout.vue`：
- 顶部导航栏（NavComponent）
- 内容区（router-view）
- 根据用户角色动态渲染导航项

**检查点**：
- [ ] 导航栏滑动效果正常
- [ ] 角色切换显示不同导航项
- [ ] 内容区切换正常（无动画，直接替换）
- [ ] 用户确认后再进入阶段 3

---

### 阶段 3：用户认证

**目标**：实现登录、JWT 认证、角色路由。

**后端**：
1. 创建 `users` 表（id, username, password_hash, role, name, created_at）
2. 实现 `/api/v1/auth/login` 接口（bcrypt 验证，返回 JWT）
3. 实现 `/api/v1/auth/me` 接口（获取当前用户信息）
4. 创建种子数据：一个管理员 admin/admin123

**前端**：
1. 登录页面（先出一个简单的登录页 HTML 原型让用户确认样式）
2. Token 存储和请求拦截器
3. 路由守卫：未登录跳转登录页，登录后根据 role 跳转对应页面

**检查点**：
- [ ] 能用 admin/admin123 登录
- [ ] 登录后正确跳转到管理端首页
- [ ] 刷新页面保持登录状态
- [ ] 用户确认后再进入阶段 4

### 阶段 4：核心功能模块（逐个实现）

> **重要**：以下每个子模块都是独立阶段，完成一个、用户确认一个，再进入下一个。不要并行开发。

#### 4.1 管理端 — 用户管理（最先做，因为其他模块依赖用户数据）

- 数据库重构：users 单表 → staff + students 双表
- 教职工（管理员+教师）CRUD：/api/v1/users/staff/*
- 学生 CRUD：/api/v1/users/students/*
- 学号规则 SS GG CC NN（学部2+年级2+班级2+序号2）
- 侧边栏 50%/50% 固定分区（教职工 + 学生）
- 居中 Modal 弹窗，分页折叠省略

**交互要求**：先出一个学生列表页的 HTML 原型（含表格样式、操作按钮、分页），用户确认后再实现。

#### 4.2 学生端 — 学习模块

- 成绩查询（表格 + 趋势折线图）
- 作业列表与提交
- 考试安排查看
- 错题本
- 课表展示

#### 4.3 教师端 — 教学模块

- 成绩录入（批量录入，支持 Excel 导入）
- 成绩统计（分布图、对比图）
- 作业布置与批改
- 出卷组卷

#### 4.4 考勤模块（学生+教师共用）

- 教师端：课堂点名
- 学生端：查看考勤记录、提交请假
- 管理端：考勤统计

#### 4.5 图书系统

- 图书检索与浏览
- 借阅/归还流程
- 管理员：图书入库、编目、统计
- 阅读推广：读书笔记、积分

#### 4.6 后勤系统

- 餐饮消费：学生查看每日/每顿/每月账单，教师查看全班汇总
- 宿舍管理：床位分配、空调电费查看与充值、报修
- 管理端：餐厅窗口管理、后勤数据大屏

#### 4.7 管理端 — 教务管理

- 排课系统
- 考试管理
- 学期管理

#### 4.8 管理端 — 数据总览

- 数据大屏（全校关键指标可视化）
- 统计报表（多维度导出）

**每个子模块的流程**：
1. 列出该模块要实现的功能
2. 出 HTML 原型（关键页面的视觉效果）
3. 用户确认原型
4. 实现后端 API
5. 实现前端页面
6. 联调测试
7. 用户验收确认
8. 进入下一个子模块

## 原型制作规范

当需要用户确认视觉设计时，生成独立的 HTML 文件，要求：

1. **单文件**：HTML + CSS + JS 全内联，不依赖外部文件
2. **使用真实配色**：用阶段 1 确定的设计 token
3. **可交互**：按钮可以点击、表格可以翻页、表单可以填写
4. **响应式**：至少适配桌面端宽度
5. **放真实数据**：用合理的假数据，不要用 "Lorem ipsum" 或 "测试1/测试2"
6. **说明用途**：文件开头注释说明"这是 XX 页面的视觉原型，用于确认 XX 设计"

## 代码规范

### 前端
- 组件用 `<script setup lang="ts">` + Composition API
- 样式用 `<style scoped>`，全局变量在 `variables.scss`
- 路由按模块组织：`/student/learn/*`、`/teacher/teach/*`、`/admin/*`
- API 请求统一封装在 `src/api/` 目录

### 后端
- 按模块组织 router：`routers/auth.py`、`routers/students.py` 等
- 数据库模型在 `models/`，Pydantic schema 在 `schemas/`
- 所有接口加类型注解，确保 Swagger 文档完整
- 密码用 bcrypt（不用 passlib，避免兼容性问题）

### 数据库
- 开发用 SQLite（`app.db`），生产用 MySQL
- 用 Alembic 管理迁移
- 表名用复数形式：`users`、`classes`、`scores`

## 禁止事项

- ❌ 禁止一次性写完所有代码再测试
- ❌ 禁止在用户没确认配色的情况下就用默认蓝色
- ❌ 禁止跳过原型直接写 Vue 组件
- ❌ 禁止并行开发多个模块
- ❌ 禁止在 AI 模块之前就把 AI 相关代码写进去
- ❌ 禁止用 `passlib`（与 bcrypt >= 5.0 不兼容，直接用 `bcrypt` 库）
- ❌ 禁止硬编码密码或密钥到代码中（开发阶段的默认值除外，但需注释说明）
