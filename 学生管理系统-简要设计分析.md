## 智慧学生管理系统 — 简要设计分析

### 一、项目概述

本项目基于 **Java SE + JDBC + MySQL** 技术栈，采用经典的分层架构（MVC 思想），实现一个功能完善的学生信息管理系统。系统面向高校/中学场景，支持管理员、教师、学生三种角色，覆盖从基础信息维护到成绩统计分析的完整业务流程。项目独立使用名为 **`student_manager_db`** 的数据库，与其它系统的 `student_manager` 数据库进行物理隔离。

**技术选型说明：** Java SE 作为运行环境，JDBC 负责数据库访问层，MySQL 提供持久化存储。为了实现数据库分离，项目专门创建了 `student_manager_db` 数据库，且表结构采用单数命名法（如 `student`、`class`、`score`），避免与其它使用复数命名法（如 `students`、`classes`）的项目发生冲突。

### 二、系统架构

```
┌─────────────────────────────────────────────┐
│              View 层（终端交互）                   │
│   菜单驱动、表格输出、输入校验、格式化打印            │
├─────────────────────────────────────────────┤
│              Controller 层                    │
│   业务调度、参数校验、异常处理、角色权限判断        │
├─────────────────────────────────────────────┤
│              Service 层                       │
│   核心业务逻辑、事务控制、数据组装               │
├─────────────────────────────────────────────┤
│              DAO 层（JDBC）                    │
│   数据库连接池、CRUD 封装、SQL 执行             │
├─────────────────────────────────────────────┤
│              MySQL 数据库                      │
│   数据表、视图、索引、外键约束                   │
└─────────────────────────────────────────────┘
```

项目采用 **DBCP 连接池**（或自实现简易连接池）避免频繁创建/销毁连接，所有数据库操作通过 DAO 接口隔离，便于维护和测试。

### 三、功能模块设计

#### 模块 1：用户与权限管理

这是整个系统的入口和安全基座。系统支持三种角色——管理员拥有全部权限，教师可管理课程/成绩/考勤，学生只能查看自己的信息。

| 功能点 | 说明 |
|---|---|
| 登录/注销 | 账号密码验证，密码使用 MD5 加盐存储，登录失败次数限制（5次锁定） |
| 密码修改 | 用户可修改自己的密码，管理员可重置他人密码 |
| 角色权限 | 基于角色的菜单可见性控制，不同角色进入不同主界面 |
| 账户管理 | 管理员可增删改查所有用户账户，支持启用/禁用 |

#### 模块 2：学生信息管理

系统的核心数据模块，所有其他模块都围绕学生展开。

| 功能点 | 说明 |
|---|---|
| 学生档案 CRUD | 学号、姓名、性别、出生日期、手机号、邮箱、地址、入学日期等 |
| 班级归属 | 每个学生关联一个班级，支持转班操作 |
| 照片管理 | 可选，存储照片路径（不存二进制，避免数据库膨胀） |
| 多条件查询 | 按学号/姓名/班级/专业模糊搜索，支持组合筛选 |
| 分页浏览 | 数据量大时分页展示，每页 10/20/50 条可选 |
| 批量导入/导出 | 从 CSV 文件批量导入学生，支持导出为 CSV |

#### 模块 3：教师信息管理

| 功能点 | 说明 |
|---|---|
| 教师档案 CRUD | 工号、姓名、性别、手机号、邮箱、职称、所属院系 |
| 授课关联 | 教师与课程多对多关联，一个教师可教多门课 |
| 班主任指定 | 一个教师可担任某班级的班主任 |

#### 模块 4：班级管理

| 功能点 | 说明 |
|---|---|
| 班级 CRUD | 班级编号、班级名称、所属专业、入学年份、学制 |

#### 模块 5：课程管理

| 功能点 | 说明 |
|---|---|
| 课程 CRUD | 课程编号、课程名、学分、学时、课程类型（必修/选修/公选） |

| 选课管理 | 选修课的选课/退课，含人数上限控制 |

这是业务复杂度最高的模块，涉及录入、修改、统计、排名等多个维度。

| 功能点 | 说明 |

### 四、数据库设计

#### 核心实体关系

```
用户(user) ──1:1──> 学生(student) / 教师(teacher)
院系(department) ──1:N──> 专业(major)
专业(major) ──1:N──> 班级(class)
班级(class) ──1:N──> 学生(student)
教师(teacher) ──M:N──> 课程(course) ──通过 teaching_plan──> 班级(class)
课程(course) ──M:N──> 学生(student) ──通过 score──> 成绩
考勤(attendance) ──N:1──> 学生 + 课程 + 日期
公告(notice) ──N:1──> 发布者(user)
```

#### 主要数据表设计

**user 用户表：** id, username, password_hash, salt, role(ENUM), real_name, status, created_at, updated_at

**student 学生表：** id, student_no(UK), name, gender, birth_date, phone, email, address, enroll_date, class_id(FK), photo_path, status, created_at

**teacher 教师表：** id, teacher_no(UK), name, gender, phone, email, title, department_id(FK), status, created_at

**department 院系表：** id, dept_no(UK), dept_name

**major 专业表：** id, major_no(UK), major_name, dept_id(FK), duration_years

**class 班级表：** id, class_no(UK), class_name, major_id(FK), enroll_year, head_teacher_id(FK→teacher)

**course 课程表：** id, course_no(UK), course_name, credit, hours, type(必修/选修/公选)

**teaching_plan 教学计划表：** id, course_id(FK), teacher_id(FK), class_id(FK), semester, 定义"谁在哪个学期教哪个班的哪门课"

**score 成绩表：** id, teaching_plan_id(FK), student_id(FK), score, grade_level, exam_type(平时/期中/期末), created_at, updated_at

**attendance 考勤表：** id, student_id(FK), teaching_plan_id(FK), attend_date, status(出勤/迟到/早退/旷课/请假), remark

**schedule 课表表：** id, teaching_plan_id(FK), day_of_week, section_start, section_end, classroom, campus

**notice 公告表：** id, title, content, publisher_id(FK→user), target_role, is_top, status, created_at

**leave_request 请假申请表：** id, student_id(FK), start_date, end_date, reason, status(待审批/已批准/已拒绝), approver_id, approve_time, remark

### 六、关键技术要点

**数据库连接池：** 手写简易连接池（实现 `DataSource` 接口），预创建 N 个 Connection 放入队列，使用时取出、用完归还。支持最大连接数限制和超时回收。这是面试高频考点，建议自己实现而非直接用 DBCP。

**DAO 模式：** 每个实体对应一个 DAO 接口和一个 JDBC 实现类。所有 SQL 使用 `PreparedStatement` 防止注入。ResultSet 通过工具方法统一映射为实体对象（简易 ORM 思想）。

**事务管理：** Service 层通过 `Connection` 手动控制事务（`setAutoCommit(false)` / `commit()` / `rollback()`），保证业务操作的原子性，比如录入成绩时批量插入多条记录。

**密码安全：** 用户密码不存明文，使用 `MD5(盐 + 密码)` 存储。盐值随机生成并随密码一起存入 user 表。

**分页查询：** SQL 层面使用 `LIMIT offset, size` 实现物理分页，DAO 层封装为通用分页方法，返回 `Page<T>` 对象（含数据列表、总数、当前页、总页数）。

**终端交互（核心）：** 系统采用纯终端交互模式，所有操作通过 `Scanner` + `System.out` 完成。交互设计遵循"菜单驱动 + 输入校验 + 友好反馈"的模式，详见下方终端交互设计章节。

**异常体系：** 自定义业务异常（如 `StudentNotFoundException`、`DuplicateStudentNoException`），DAO 层抛出 `SQLException` 的包装异常，Service 层捕获并转换为业务异常，Controller 层统一处理并在终端展示友好提示。

#### 7.3 输入处理与容错

所有用户输入都经过校验，非法输入不会导致程序崩溃，而是提示重新输入：

```
请输入学生姓名 > abc123
[错误] 姓名只能包含中文和字母，请重新输入 >

请输入分数(0-100) > 150
[错误] 分数必须在 0-100 之间，请重新输入 >

确认删除该学生？(y/n) > x
[错误] 请输入 y 或 n >
```

关键实现要点：使用 `Scanner.nextLine()` 统一读取整行输入，再通过 `try-catch` 和正则校验进行类型转换和格式检查。数字输入封装为 `ConsoleUtil.readInt(prompt, min, max)` 方法，支持范围限制和循环重试。

#### 7.6 终端交互注意事项

**中文对齐问题：** 中文字符占两个显示宽度，直接 `String.format` 会导致表格错位。解决方案是在计算列宽时，将中文字符按 2 个宽度计算（遍历 char 判断 `Character.UnicodeScript` 或直接用正则匹配中文范围），填充空格时减去中文字符数。

**大量数据输出：** 列表数据超过一屏时，使用分页展示而非一次性输出。每页显示 10/20 条，底部提供翻页操作提示。

**操作确认：** 删除、批量修改等不可逆操作前，必须显示操作摘要并要求 `y/n` 确认，防止误操作。

**输入历史：** `Scanner` 本身不支持上下翻阅历史，但可以在 Controller 层维护一个 `List<String>` 记录最近输入，供用户参考（可选功能）。

### 九、开发阶段建议

**第一阶段（基础框架）：** 搭建项目结构，实现数据库连接池和 DAO 基类，完成用户登录和权限模块，建好全部数据表并灌入测试数据。

**第二阶段（核心业务）：** 完成学生/教师/班级/课程的 CRUD，实现多条件查询和分页，完成成绩的录入、修改、查询和统计。

**第三阶段（扩展功能）：** 实现考勤管理、课表排课、请假审批、公告系统。

**第四阶段（完善优化）：** 实现数据导入导出（CSV），完善统计报表，添加数据校验和异常处理，优化交互体验，编写 README 和使用文档。

---

### 十、测试数据规划

系统初始化时应预置一批有代表性的测试数据：3-5 个院系、8-10 个专业、10+ 个班级、50+ 名学生、10+ 名教师、15+ 门课程、若干学期的成绩和考勤记录。这些数据要能支撑统计分析模块展示出有意义的结果（如分数分布、排名、趋势变化等）。

---

> **总结：** 本系统以 Java SE + JDBC + MySQL 为基础，采用分层架构，涵盖用户权限、学生/教师/班级/课程管理、成绩管理、考勤管理、课表管理、公告通知、数据报表共 10 个功能模块。设计上注重代码规范性和可维护性，适合作为 Java 综合实践项目。
